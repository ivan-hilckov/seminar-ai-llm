- Mono-подпись с путём: `Settings → Data controls → Improve the model for everyone`

## 2. Технический фон

### Что технически делает этот toggle
Включает или выключает использование ваших чатов в **обучающем датасете будущих релизов** модели. Это не влияет на:
- Логи провайдера (запросы всё равно сохраняются у DeepSeek/OpenAI/etc. на серверах)
- Возможность поддержки (сотрудники провайдера могут читать ваши чаты по запросу, если, например, идёт расследование инцидента)
- Текущую модель (она уже заморожена и не учится в моменте — это слайд 36)

Toggle влияет на одно: попадут ли ваши чаты в датасет для тренировки следующей версии модели.

### Где этот toggle у разных провайдеров (для подачи)

**ChatGPT (OpenAI):**
- Settings → Data Controls → Improve the model for everyone → OFF
- Альтернатива: Temporary Chat (значок в правом верхнем углу) — для разового чата без сохранения

**DeepSeek:**
- Боковая панель → три точки рядом с именем → Settings → Data controls → Improve the model for everyone → OFF
- В мобильном приложении путь такой же

**Claude (Anthropic):**
- По умолчанию данные **не** используются для обучения ни в чате, ни в API. Toggle для отключения не нужен.

**Gemini (Google):**
- gemini.google.com → Activity → Off
- В бесплатном тарифе ответы могут проверять разметчики; в платных — нет

**Yandex GPT:**
- В кабинете Yandex Cloud у юридических лиц можно настроить политику обработки. У частных пользователей через приложение Yandex — настройки обработки данных в общих настройках Yandex-аккаунта.

### Что toggle НЕ делает
- Не удаляет уже отправленные запросы из логов (для этого надо отдельно писать на privacy@<провайдер>)
- Не отключает сохранение истории чатов (она по-прежнему доступна вам в интерфейсе)
- Не делает запрос анонимным — провайдер всё равно видит, что это запрос от вашего аккаунта

### Почему у DeepSeek и ChatGPT текст toggle одинаковый
Формулировка «Improve the model for everyone» — это стандартный паттерн UX, который OpenAI ввёл первым. Многие провайдеры (включая DeepSeek) переняли его дословно, потому что он уже понятен пользователям.

## 6. На что обратить внимание

- **Темп:** средний. Это инструкция, она должна остаться в памяти.
- **Не использовать слова «страх», «бояться», «слив»** — это нейтральная инженерная настройка
- **Подчеркнуть универсальность:** «не только DeepSeek — везде есть такой переключатель»
- **Не обещать абсолютную приватность** через этот toggle: он защищает от попадания в обучающий датасет, не от логирования
- **Не углубляться в политики провайдеров.** Если спросят про конкретный — назвать путь и идти дальше

## 8. Возможные вопросы из зала

**«А Claude — где у него этот toggle?»**
> У Anthropic данные чата не используются для обучения по умолчанию — отдельный toggle для отключения не нужен.

**«Если я выключил — мои предыдущие чаты тоже не попадут в обучение?»**
> Toggle действует на будущее. Чтобы убрать уже отправленные данные из обучающих наборов, нужно отдельно писать в саппорт провайдера — например, у DeepSeek это privacy@deepseek.com.

**«А логи всё равно остаются?»**
> Да. Этот toggle отвечает только за обучение. Логи провайдера — отдельная история; они хранятся обычно 30 дней, у разных провайдеров политики разные.

**«Можно вообще не сохранять чат в истории?»**
> У ChatGPT есть режим Temporary Chat — значок сверху справа. Чат не сохраняется в истории и не идёт в обучение. У DeepSeek аналогичной кнопки одного клика нет.

**«А корпоративная подписка — там по-другому?»**
> Да. В платных корпоративных тарифах (ChatGPT Team, Enterprise; Claude for Work) данные по умолчанию не используются для обучения — это часть условий контракта.

**«Если я через API работаю?»**
> В API-режиме у OpenAI, Anthropic и большинства провайдеров данные **не** используются для обучения по умолчанию. Это разница между чатом и API.

**«Где у Yandex GPT этот переключатель?»**
> В персональном аккаунте Yandex — в настройках обработки данных. В корпоративном Yandex Cloud — отдельная политика в кабинете администратора.

## 9. Провокационные вопросы

**«Они всё равно врут и используют наши данные.»**
> Технически проверить невозможно. Но крупные провайдеры под жёстким регуляторным контролем — особенно в ЕС после AI Act. Для рабочих задач с чувствительными данными решение надёжнее: локальная модель, как обсуждали на предыдущем слайде.

*Не вступать в спор о доверии — переводить на инженерное решение.*

**«А китайская модель — они же там вообще всё читают.»**
> Юрисдикция провайдера — отдельный вопрос. Для чувствительных задач — локальная модель закрывает его полностью. Toggle в облачной модели — это минимальный уровень защиты, не максимальный.
